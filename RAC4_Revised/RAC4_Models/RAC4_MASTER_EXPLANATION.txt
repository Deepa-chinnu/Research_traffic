PART 1: RESEARCH OBJECTIVE
================================================================================

WHAT IS THIS RESEARCH ABOUT?
    We are predicting road traffic volume (number of vehicles per day)
    in Bangalore city using machine learning. Given information like
    average speed, congestion level, road capacity, weather, and time,
    can we predict how many vehicles will be on the road?

WHY IS THIS IMPORTANT?
    - Bangalore has severe traffic congestion (ranked among worst in world)
    - Accurate prediction helps traffic management systems plan better
    - Can reduce commute time, fuel consumption, and pollution
    - Helps city planners decide where to build new roads or signals

WHAT DID WE DO?
    1. Collected Bangalore traffic data (8,936 records from 2022-2024)
    2. Cleaned and prepared the data (preprocessing)
    3. Trained 10 different machine learning models (27 configurations)
    4. Compared all models to find the best one
    5. Analysed how each model can be improved
    6. Studied temporal patterns (when does traffic increase/decrease)

PART 3: DATA PREPROCESSING (Every Step Explained)
================================================================================

WHY DO WE NEED PREPROCESSING?
    Raw data is messy. Machine learning models need clean, numeric data.
    Preprocessing transforms raw data into a format that models can learn from.
    Without proper preprocessing, models give poor or misleading results.

STEP-BY-STEP PREPROCESSING:

STEP 3.1: LOADING THE DATA
    - Read the CSV file into a pandas DataFrame
    - Verified: 8,936 rows, 16 columns loaded successfully
    - File: step1_data_preprocessing.py
    - Output: Printed data shape and column names

STEP 3.2: MISSING VALUE CHECK
    WHY: If data has missing values, models cannot process those rows.
         We need to either fill them or remove them.
    RESULT: No missing values found in any column (all 8,936 rows complete).
    GRAPH: missing_values.png shows green bars for all columns (no missing data).
    HOW TO EXPLAIN: "We checked every column for missing values. The dataset
                     is complete with zero missing values across all 16 columns."

STEP 3.3: DATA TYPE VERIFICATION
    WHY: Some numbers may be stored as text. Models need actual numbers.
    WHAT WE DID: Converted all numeric columns using pd.to_numeric().
                 Any value that cannot be converted is marked as invalid
                 and removed.
    RESULT: All 8,936 rows survived (no invalid values found).

STEP 3.4: FEATURE ENGINEERING - TEMPORAL FEATURES
    WHY: The Date column contains hidden patterns. Traffic is different
         on Monday vs Sunday, January vs July. We extract these patterns.

    From the Date column, we created 7 new features:

    Feature         Description                     Why it matters
    -------         -----------                     ---------------
    Year            2022, 2023, or 2024             Year-over-year trends
    Month           1 to 12                         Seasonal patterns (monsoon, festivals)
    Day             1 to 31                         Day-of-month patterns
    DayOfWeek       0=Monday to 6=Sunday            Weekday vs Weekend patterns
    WeekOfYear      1 to 52                         Weekly cycles
    IsWeekend       0=Weekday, 1=Weekend            Binary weekday/weekend flag
    Quarter         1=Jan-Mar, 2=Apr-Jun, etc.      Seasonal quarter

    HOW TO EXPLAIN: "We extracted 7 temporal features from the Date column
                     because traffic follows time-based patterns. For example,
                     traffic is heavier on weekdays due to office commutes,
                     and lighter during monsoon season due to heavy rains."

STEP 3.5: ENCODING CATEGORICAL VARIABLES
    WHY: Models only understand numbers, not text like "Indiranagar" or "Rain".
         We convert text categories into numbers.

    WHAT WE DID:

    a) Weather Conditions -> One-Hot Encoding (dummy variables)
       Clear is the base category (dropped to avoid multicollinearity)
       Created 4 binary columns: Weather_Fog, Weather_Overcast,
                                  Weather_Rain, Weather_Windy
       Example: If weather is "Rain", then Weather_Rain=1, rest=0

       WHY one-hot encoding? Because weather categories have no natural order.
       "Rain" is not greater than "Clear". One-hot encoding treats them equally.

    b) Roadwork Activity -> Binary Encoding
       Yes = 1, No = 0
       WHY binary? Because there are only 2 categories.

    c) Area Name -> Label Encoding
       Each area gets a unique number: Indiranagar=0, Whitefield=1, etc.
       WHY label encoding? To include spatial information without creating
       too many dummy columns (8 areas = 8 columns with one-hot).

    d) Road/Intersection Name -> Label Encoding
       Each road gets a unique number (16 unique roads).

    HOW TO EXPLAIN: "We converted all text data into numbers. Weather uses
                     one-hot encoding because weather types have no order.
                     Areas and roads use label encoding to keep the feature
                     count manageable."

STEP 3.6: CORRELATION ANALYSIS

           Dark red = strong positive, Dark blue = strong negative, White = no correlation.

STEP 3.7: TRAIN-TEST SPLIT
    WHY: We split data into two parts:
         - Training set (70%): Model learns patterns from this data
         - Testing set (30%): We evaluate the model on this unseen data
         If we test on the same data we trained on, the model just memorises
         the answers. Testing on unseen data tells us if the model truly learned.

    RESULT:
         Training set: 6,255 samples (70.0%)
         Testing set : 2,681 samples (30.0%)
         Random state: 42 (fixed seed ensures same split every time we run)

    GRAPH: train_test_split.png shows the distribution of Traffic Volume in
           both sets. They should look similar (representative split).

    HOW TO EXPLAIN: "We used 70% of data for training and kept 30% completely
                     hidden from the model during training. This unseen 30%
                     is used to test if the model can predict accurately on
                     new data it has never seen before."

STEP 3.8: FEATURE SCALING (StandardScaler)
    WHY: Some features have values 0-10 (Incident Reports), while others
         have values 4,000-72,000 (Traffic Volume). Models like Linear
         Regression, Ridge, SVR, and KNN are sensitive to these differences.
         Scaling makes all features have mean=0 and std=1.

    IMPORTANT RULE: Scaler is fitted ONLY on training data. Test data is
    transformed using training parameters. This prevents data leakage.

    WHICH MODELS NEED SCALING:
         Need scaling: Linear Regression, Ridge, Lasso, ElasticNet, SVR, KNN
         Do NOT need:  Decision Tree, Random Forest, Extra Trees, Gradient Boosting

    HOW TO EXPLAIN: "We standardised features so that all have equal scale.
                     The scaler was fitted only on training data to prevent
                     information leakage from the test set."

FINAL FEATURE COUNT: 24 features total
    10 numeric (original) + 7 temporal (engineered) + 3 encoded + 4 weather dummies

----------------------------------------------------------------------
MODEL 1: LINEAR REGRESSION
----------------------------------------------------------------------

WHAT IS IT?
    The simplest model. It draws a straight line (or hyperplane in multiple
    dimensions) through the data.

    Formula: y = b0 + b1*x1 + b2*x2 + ... + b24*x24

    Where y = predicted Traffic Volume
          x1 to x24 = our 24 features
          b0 = intercept (baseline value)
          b1 to b24 = coefficients (how much each feature contributes)

HOW DOES IT WORK?
    1. Starts with random coefficients
    2. For each training sample, calculates the prediction
    3. Measures the error (difference between prediction and actual)
    4. Adjusts coefficients to minimize total error (sum of squared errors)
    5. Repeats until optimal coefficients are found

WHY DID WE USE IT?
    - Baseline model - simplest possible approach
    - Very fast to train (instant)
    - Coefficients are interpretable (tells us which features matter)
    - If linear regression works well, the problem might be inherently linear

RESULT: Test R2 = 1.0000, RMSE = 0.0
    (Perfect score due to Environmental Impact feature - see Part 9)

LIMITATION: Cannot capture nonlinear relationships. If traffic volume has
            complex interactions between features, linear regression misses them.

FILES GENERATED: outputs/03_Models/01_LinearRegression/
    - predicted_vs_actual.png : Points should lie on diagonal line
    - residual_analysis.png  : Residuals should be randomly scattered around 0
    - coefficients.png       : Bar chart showing which features have positive/negative effect
    - report.txt             : Detailed text report

----------------------------------------------------------------------
MODEL 2: RIDGE REGRESSION
----------------------------------------------------------------------

WHAT IS IT?
    Linear Regression with L2 regularization. It adds a penalty for large
    coefficients to prevent overfitting.

    Formula: Same as Linear Regression + lambda * sum(coefficient^2)

    The lambda (alpha) parameter controls how much penalty is applied:
    - alpha = 0.1  : Light penalty (close to basic Linear Regression)
    - alpha = 1.0  : Moderate penalty
    - alpha = 10.0 : Strong penalty (coefficients shrink towards zero)

WHY IS IT BETTER THAN LINEAR REGRESSION?
    - Prevents overfitting by keeping coefficients small
    - Handles multicollinearity (when features are correlated with each other)
    - More stable predictions
    - Still very fast and interpretable

WHY DID WE TEST 3 ALPHA VALUES?
    To show how different regularization strengths affect performance.
    This is called "hyperparameter tuning".

RESULTS:
    alpha=0.1  : R2 = 1.0000, RMSE = 0.41
    alpha=1.0  : R2 = 1.0000, RMSE = 4.12
    alpha=10.0 : R2 = 0.9999, RMSE = 40.82

    As alpha increases, RMSE increases slightly (model is more conservative).

FILES: outputs/03_Models/02_Ridge/alpha_0.1/, alpha_1.0/, alpha_10.0/

----------------------------------------------------------------------
MODEL 3: LASSO REGRESSION
----------------------------------------------------------------------

WHAT IS IT?
    Linear Regression with L1 regularization. Unlike Ridge, Lasso can
    set some coefficients to EXACTLY ZERO, performing automatic feature selection.

    Formula: Same as Linear Regression + lambda * sum(|coefficient|)

WHY IS LASSO SPECIAL?
    - It SELECTS features automatically
    - Features with zero coefficient are deemed unimportant
    - Useful for understanding which features truly matter
    - Simpler model (fewer active features)

DIFFERENCE FROM RIDGE:
    Ridge: Shrinks all coefficients but keeps all features
    Lasso: Can completely eliminate unimportant features

RESULTS:
    alpha=0.001: R2 = 1.0000, RMSE = 4.57 (all features kept)
    alpha=0.01 : R2 = 1.0000, RMSE = 4.48
    alpha=0.1  : R2 = 1.0000, RMSE = 3.61
    alpha=1.0  : R2 = 1.0000, RMSE = 1.15 (some features may be eliminated)

FILES: outputs/03_Models/03_Lasso/alpha_*/

----------------------------------------------------------------------
MODEL 4: ELASTICNET
----------------------------------------------------------------------

WHAT IS IT?
    Combination of Ridge (L2) and Lasso (L1) regularization.

    Formula: Linear Regression + lambda1*sum(|coef|) + lambda2*sum(coef^2)

    l1_ratio = 0.5 means 50% L1 (Lasso) + 50% L2 (Ridge)

WHY USE BOTH?
    - Gets benefits of both: feature selection (Lasso) + stability (Ridge)
    - More robust than pure Lasso when features are correlated
    - Better when you're unsure whether to use Ridge or Lasso

RESULTS:
    alpha=0.001: R2 = 1.0000, RMSE = 12.87
    alpha=0.01 : R2 = 0.9999, RMSE = 124.75
    alpha=0.1  : R2 = 0.9941, RMSE = 1001.02

    Higher alpha = stronger regularization = more features eliminated = higher error

FILES: outputs/03_Models/04_ElasticNet/alpha_*/

----------------------------------------------------------------------
MODEL 5: SUPPORT VECTOR REGRESSION (SVR)
----------------------------------------------------------------------

WHAT IS IT?
    SVR finds a function that fits most data points within a margin (epsilon).
    Points outside this margin are penalised.

TWO KERNELS TESTED:
    1. RBF (Radial Basis Function) Kernel:
       - Maps data to higher dimensions to find nonlinear patterns
       - Can capture complex relationships
       - Slower, requires good hyperparameter tuning

    2. Linear Kernel:
       - Similar to Linear Regression with regularization
       - Faster but limited to linear relationships

WHAT IS A KERNEL?
    A kernel is a function that transforms data into a higher-dimensional
    space where patterns might be easier to find. Think of it as looking
    at data from a different angle.

    Example: If data forms a circle (nonlinear), lifting it into 3D might
    make it separable by a flat plane (linear in 3D).

RESULTS:
    RBF kernel    : R2 = 0.0113, RMSE = 12,950.9 (VERY POOR - needs tuning)
    Linear kernel : R2 = 0.7747, RMSE = 6,181.9   (Moderate)

    WHY IS SVR-RBF SO BAD?
    - Default C=1.0 is too low for this dataset
    - SVR-RBF is very sensitive to the C parameter
    - With C=100, it improves to R2=0.79 (shown in improvement analysis)
    - The data has 8,936 samples - SVR scales poorly with large datasets

FILES: outputs/03_Models/05_SVR/kernel_rbf/, kernel_linear/

----------------------------------------------------------------------
MODEL 6: K-NEAREST NEIGHBORS (KNN)
----------------------------------------------------------------------

WHAT IS IT?
    The simplest nonlinear model. To predict traffic volume for a new record,
    it finds the K most similar records in the training data and averages
    their traffic volumes.

HOW DOES IT WORK?
    1. Receives a new data point to predict
    2. Calculates distance to ALL training data points (using Euclidean distance)
    3. Finds the K closest (nearest) neighbours
    4. Averages their Traffic Volume values
    5. That average is the prediction

WHAT IS K?
    K = number of neighbours to consider
    - K=3  : Uses 3 nearest neighbours (sensitive to local noise)
    - K=5  : Uses 5 nearest neighbours (moderate)
    - K=11 : Uses 11 nearest neighbours (smoother, more robust)

WHY DIFFERENT K VALUES?
    - Small K: More sensitive to local patterns but also to noise (overfitting risk)
    - Large K: Smoother predictions but may miss local variations (underfitting risk)
    - Finding optimal K is done via the "elbow method" (shown in improvements)

RESULTS:
    K=3  : R2 = 0.8429, RMSE = 5,162.9  (slight overfitting)
    K=5  : R2 = 0.8576, RMSE = 4,915.4  (slight overfitting)
    K=11 : R2 = 0.8625, RMSE = 4,828.9  (good generalization)

    WHY DOES KNN PERFORM WORSE THAN TREE MODELS?
    - KNN uses distance, so all features contribute equally
    - In 24 dimensions, distance becomes less meaningful (curse of dimensionality)
    - KNN cannot learn which features are important (no feature selection)

FILES: outputs/03_Models/06_KNN/k_3/, k_5/, k_11/

----------------------------------------------------------------------
MODEL 7: DECISION TREE
----------------------------------------------------------------------

WHAT IS IT?
    A tree-shaped model that makes predictions by asking a series of
    yes/no questions about the features.

HOW DOES IT WORK?
    Starting from the root (top):
    1. Ask: "Is Environmental Impact <= 113.09?"
       - If YES: Go left branch
       - If NO : Go right branch
    2. At each branch, ask another question
    3. Continue until reaching a leaf node (final answer)
    4. The leaf node gives the predicted Traffic Volume

    The tree learns which questions to ask and what thresholds to use
    by finding the splits that minimize prediction error.

WHAT IS max_depth?
    Maximum number of levels in the tree:
    - max_depth=6     : Tree can ask 6 questions deep (127 nodes, 64 leaves)
    - max_depth=12    : Tree can ask 12 questions deep (6,913 nodes, 3,457 leaves)
    - max_depth=None  : No limit (11,705 nodes, 5,853 leaves) - RISK OF OVERFITTING

    WHY LIMIT DEPTH?
    - Unlimited depth: Tree memorises every training sample (100% training accuracy)
      but may fail on new data (overfitting)
    - Limited depth: Tree learns general patterns instead of memorising

RESULTS:
    max_depth=6     : R2 = 0.9996, RMSE = 268.7  (simpler tree, slightly less accurate)
    max_depth=12    : R2 = 1.0000, RMSE = 38.6    (good balance)
    max_depth=None  : R2 = 1.0000, RMSE = 32.7    (most accurate but risk of overfitting)

WHERE IS THE TREE VISUALIZATION?

          This shows the actual tree with nodes, splits, and leaf values.

    FILE: outputs/03_Models/07_DecisionTree/max_depth_12/tree_rules.txt
          This shows the tree as text rules:
          "If Environmental Impact <= 113.09 AND Environmental Impact <= 89.64
           THEN Traffic Volume = [predicted value]"

    FILE: outputs/03_Models/07_DecisionTree/max_depth_12/feature_importance.png
          Shows which features the tree uses most for splitting.

HOW TO READ THE TREE DIAGRAM:
    - Each box is a NODE
    - Top line: the question (e.g., "Environmental Impact <= 113.09")
    - "samples": how many training records reach this node
    - "value": the average Traffic Volume at this node
    - Colour: darker = more certain (lower error)
    - Left branch: condition is TRUE
    - Right branch: condition is FALSE

----------------------------------------------------------------------
MODEL 8: RANDOM FOREST
----------------------------------------------------------------------

WHAT IS IT?
    An ensemble (collection) of multiple Decision Trees. Instead of one
    tree, we build many trees and average their predictions.

HOW DOES IT WORK?
    1. Create N different datasets by randomly sampling from training data
       (sampling WITH replacement - same record can appear multiple times)
    2. Build a Decision Tree on each sampled dataset
    3. At each split, only consider a RANDOM SUBSET of features
       (this ensures trees are different from each other)
    4. For prediction: each tree gives its answer, take the AVERAGE

WHY IS IT BETTER THAN A SINGLE DECISION TREE?
    - Single tree: might overfit (memorise noise in the data)
    - Random Forest: averaging many trees cancels out individual errors
    - This is called "bagging" (Bootstrap AGGregatING)
    - Each tree sees different data and features -> diverse predictions
    - Diverse predictions averaged together = robust final prediction

WHAT IS n_estimators?
    The number of trees in the forest:
    - 50 trees  : Faster training, slightly less accurate
    - 100 trees : Good balance (most common choice)
    - 200 trees : Slightly better but takes longer

RESULTS:
    50 trees  : R2 = 1.0000, RMSE = 38.2, Training time = 0.73s
    100 trees : R2 = 1.0000, RMSE = 34.4, Training time = 1.45s
    200 trees : R2 = 1.0000, RMSE = 31.9, Training time = 2.71s

    More trees = slightly better RMSE but slower training.
    Diminishing returns after ~200 trees.

WHERE ARE THE TREE VISUALIZATIONS?

          Shows 3 individual trees from the forest (top 3 levels).
          Each tree is DIFFERENT because it was built on different data.

          Detailed view of Tree #1 (top 5 levels).

FILES: outputs/03_Models/08_RandomForest/50_trees/, 100_trees/, 200_trees/

MODEL 9: EXTRA TREES (Extremely Randomized Trees)
----------------------------------------------------------------------

WHAT IS IT?
    Similar to Random Forest but with EVEN MORE randomization.

HOW IS IT DIFFERENT FROM RANDOM FOREST?
    Random Forest: At each split, finds the BEST split point
    Extra Trees:   At each split, uses a RANDOM split point

    This makes Extra Trees:
    - FASTER to train (no optimization at each split)
    - Sometimes more robust to noise (extra randomization helps)
    - Sometimes slightly less accurate (random splits may not be optimal)

----------------------------------------------------------------------
MODEL 10: GRADIENT BOOSTING
----------------------------------------------------------------------

WHAT IS IT?
    An ensemble that builds trees SEQUENTIALLY, where each new tree
    corrects the errors of the previous ones.

HOW IS IT DIFFERENT FROM RANDOM FOREST?
    Random Forest:   Builds trees IN PARALLEL (independently), then averages
    Gradient Boosting: Builds trees ONE BY ONE (sequentially)

    Step 1: Build Tree 1, measure errors
    Step 2: Build Tree 2 to predict those errors
    Step 3: Build Tree 3 to predict remaining errors
    ...
    Step N: Each tree fixes a little bit of the remaining error

WHAT IS learning_rate?
    How much each new tree contributes:
    - learning_rate = 0.1: Each tree contributes 10% correction
    - learning_rate = 0.01: Each tree contributes 1% (needs more trees)

    Lower learning rate + more trees = usually better but slower.

RESULTS:
    100 estimators, lr=0.1: R2 = 1.0000, RMSE = 35.4, Time = 2.77s
    200 estimators, lr=0.1: R2 = 1.0000, RMSE = 35.6, Time = 5.77s

    WHERE IS THE LEARNING CURVE?
    FILE: outputs/03_Models/10_GradientBoosting/100_estimators/learning_curve.png
    This shows how the training loss decreases as more trees are added.
    The curve should go DOWN (error decreasing) and eventually flatten
    (adding more trees doesn't help anymore).

================================================================================
PART 6: EVALUATION METRICS (What R2, RMSE, MAE, MAPE Mean)
================================================================================

We use 4 metrics to evaluate how good each model's predictions are:

METRIC 1: R2 SCORE (R-Squared / Coefficient of Determination)
    FORMULA: R2 = 1 - (sum of squared errors / total variance)
    RANGE:   0 to 1 (can be negative for very bad models)

    WHAT IT MEANS:
    - R2 = 1.0   : Perfect prediction (model explains 100% of variance)
    - R2 = 0.9   : Very good (model explains 90% of variance)
    - R2 = 0.5   : Moderate (model explains 50% of variance)
    - R2 = 0.0   : Useless (model is no better than predicting the average)
    - R2 < 0     : Terrible (model is WORSE than predicting the average)

    HOW TO EXPLAIN: "R2 tells us what percentage of traffic volume variation
                     our model can explain. R2=0.99 means 99% of the variation
                     is captured by the model."

METRIC 2: RMSE (Root Mean Squared Error)
    FORMULA: RMSE = sqrt(average of (actual - predicted)^2)
    RANGE:   0 to infinity (lower is better)

    WHAT IT MEANS:
    - RMSE is in the SAME UNITS as the target (vehicles/day)
    - RMSE = 100 means predictions are off by about 100 vehicles on average
    - RMSE gives MORE WEIGHT to large errors (because of squaring)

    HOW TO EXPLAIN: "RMSE of 35 means our predictions are off by about
                     35 vehicles per day on average. Given the mean traffic
                     volume is 29,236, this is an error of only 0.12%."

METRIC 3: MAE (Mean Absolute Error)
    FORMULA: MAE = average of |actual - predicted|
    RANGE:   0 to infinity (lower is better)

    WHAT IT MEANS:
    - Also in same units as target (vehicles/day)
    - Treats all errors equally (unlike RMSE which penalizes large errors)
    - More robust to outliers than RMSE

METRIC 4: MAPE (Mean Absolute Percentage Error)
    FORMULA: MAPE = average of |actual - predicted| / actual * 100%
    RANGE:   0% to infinity (lower is better)

    WHAT IT MEANS:
    - Error as a PERCENTAGE of the actual value
    - MAPE = 0.03% means predictions are off by 0.03% on average
    - Easier to interpret across different scales

ADDITIONAL METRICS WE USED:

METRIC 5: OVERFITTING GAP (Train R2 - Test R2)
    - If gap > 0.1  : OVERFITTING (model memorised training data)
    - If gap < 0.05 : GOOD GENERALIZATION
    - If gap < 0    : Unusual (test performance better than training)

METRIC 6: UTILITY SCORE (Custom composite metric)
    FORMULA: Utility = 0.7*R2 + 0.2*(1 - TrainTime/maxTime) + 0.1*(1 - Sensitivity)

    This balances:
    - 70% accuracy (R2)
    - 20% speed (faster training = better)
    - 10% robustness (less sensitive to parameter changes = better)

METRIC 7: 5-FOLD CROSS-VALIDATION R2
    Instead of one train-test split, we do 5 different splits:
    - Fold 1: 80% train, 20% test (different split)
    - Fold 2: 80% train, 20% test (different split)
    - ...
    - Fold 5: 80% train, 20% test (different split)
    Average R2 across all 5 folds gives more reliable estimate.

    - Skewed = data concentrated on one side
    - Red line = mean value
    WHAT TO SAY: "Most features show reasonable distributions. Average Speed
                  is roughly normal around 39 km/h."

    WHAT IT SHOWS: Model's predictions compared to real values

    - Red dashed line = perfect prediction (y = x)
    - Points ON the line = perfect predictions
    - Points FAR from line = errors
    - Left plot = Training data, Right plot = Testing data
    WHAT TO SAY: "Points closely follow the diagonal line, indicating
                  the model predicts accurately. The testing plot (right)

                  generalizes well to unseen data."

    WHAT IT SHOWS: Analysis of prediction errors (residuals = actual - predicted)
    HOW TO READ:
    - Left: Residual vs Predicted plot

      * Any pattern means the model is missing something
    - Middle: Residual histogram
      * Should be bell-shaped centered at 0
      * Skewed = model systematically over/under predicts
    - Right: Sorted residuals
      * Should be symmetric around 0
    WHAT TO SAY: "The residuals are randomly distributed around zero with
                  no visible pattern, indicating no systematic bias."

    WHAT IT SHOWS: Bar chart comparing training and testing performance
